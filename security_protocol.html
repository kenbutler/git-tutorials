Security Protocol

Security protocol, in general, supports the secure transfer of data to and from networked computers. In order to use Git with a team, it is important to understand these protocol to be able to perform server-related actions, such as cloning. While a number of security protocols exist , there are two main protocol you'll most often encounter:

HTTPS   and   SSH

The method you use when cloning down a repository will remain with that clone instance, unless you simply reclone using an alternative method or reconfigure the repository's URL on your local machine.

HTTPS

HTTPS (Hypertext Transfer Protocol Secure) is a more secure extension of HTTP that has become very popular. Modern browsers, for example, often mark websites as secure or not based on whether the site uses HTTP or HTTPS. It uses public-private key cryptography to authenticate users. When using HTTPS, Git will by default always ask for a your password for server-related actions (e.g. push, pull, and clone). It is possible, though not recommended, to have Git store your credentials for you HTTPS is easier to set up than SSH, though it is generally a bad habit to enter your password frequently.

SSH

SSH also uses public-private key cryptography to authenticate users. Unlike HTTPS though, it does not use username/password for confirming user identity. In order to create SSH keys, you'll need to either enter ssh-keygen -t rsa from the command line or use a tool such as PuTTy for Windows. This process will create public and private keys. When using SSH with Git, you'll need to ensure you upload your public (NOT private) key to your server. Your private key should never be shared. These keys can be used for multiple different servers. If you change computers, you'll need to recreate keys. This process is cheap and easy though. It's often easier to just recreate keys than resolve a protocol error when performing server-related actions (pushes, pulls, clones, etc).
