What are Binary Files?

Binary files are non-text files that require a computer program to interpret how the stored data is formatted. Git relies on text to track changes, so binary files can be problematic.

Binary Files: Engineering Examples

  • Interface Control Documents
  • Software Design Documents
  • Visio Drawings
  • SolidWorks, AutoCAD, CATIA V5 drawings or animations
  • Binary data used for input
  • Catch-all: typically any file with a *.bin extension

Binary Files: Git's Solution

Git Large File Storage (LFS)

  • Efficiently store large files
  • Support file locking to prevent simultaneous changes of binary files
  • Do NOT provide an easy way to view differences between versions of a binary file

To use Git LFS, you will need a server that can handle this -- and not all servers do!
Click the down arrow for a recommended strategy when Git LFS is not available on your server.

Binary Files: Without Git LFS

Keeping track of binary files can still be a good idea.
The following is a recommended strategy.

  • Create a separate repository to hold binary files
  • Avoid branching, as merge conflicts are probable

This helps keep the size of your code-based repository small, which is especially useful when sharing (via cloning) with customers or peers.

Git LFS: Installation

To apply Git LFS to distributed development, you will need a server that can handle it.

You can still use it locally on your machine though. For example, it may still be relevant to you if you are working on a personal project.

Git LFS must be installed on each user’s local machine using a package manager, such as Homebrew, yum, apt, etc. You will also need to add it to your $PATH variable.

Once git-lfs is on your path, execute the following:
(You should only have to do this once!) 

                
                    $ git lfs install
                    Git LFS initialized.

Also see GitHub's official documentation on installing Git LFS .

Git LFS: Repository Configuration

You will also need to update the settings of your repository on the server to allow for Git LFS.

NOTE: Applying Git LFS to existing repos will NOT retroactively apply to previous commits. This would require rewriting history.

Git LFS: Example Use Case

If we wanted to track modern PowerPoint files, we can navigate to the directory of the repository and type: 

                
                    $ git lfs track “*.pptx”
                    Tracking *.pptx

The quotes here are important, as they prevent the wildcard (*) from being expanded and creating individual entries for each *.pptx file in your current directory.

This will create (or update) a .gitattributes file, which binds tracked file patterns to the Git LFS filter. This file needs to be added and committed to the repository!

Git LFS: Example Use Case

You can check what file patterns are monitored under Git LFS for a repository like so: 

                
                    $ git lfs track
                    Listing tracked patterns
                        *.pptx (.gitattributes)
                    Listing excluded patterns
