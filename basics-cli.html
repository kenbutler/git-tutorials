Basic Commands

Command Line Interface

This section aims to provide a solid foundation for the basic commands required to use Git. To understand the applicability of the basic commands, this tutorial will follow a scenario.

You may press ESC at any point to view an overview of the tutorial. Note the arrow(s) in the bottom-right corner that controls navigation through the lesson.

Types of Repository Storage

There are two ways a Git repository can be stored and managed: locally on a personal computer or remotely on a server. Storing remotely permits distributed teamwork. It is worth noting that a locally stored repository can be moved to a server if desired.

Local Computer

Remote Server


Let's first consider local storage.

git init

Here we create a new directory called droid aimed at software for battle droid development. After making the directory, we navigate into it. Using the ls command, we see the directory is empty. Since we already have Git installed, we can run the command git init to start a local repository in the current directory. We see this present the message: Initialized empty Git repository in current directory If we examine the directories contents now, showing all hidden files using ls -a, we see that a .git directory has been added. This is a directory added by Git to allow for version control. The details of this directory are NOT covered in this tutorial set.

git init

Next we add an empty .gitignore file using the touch command. The .gitignore file essentially tells Git what files to ignore in the repository. The .gitignore file will be covered later in this lesson. It is always good practice to add a .gitignore file at the beginning of a repository.

We then add the .gitignore file to the repository and commit it with a detailed message.

git init

Git's response to this particular commit shows us that:

  • This is the root commit of the repository - (root-commit)
  • This commit was made on the master branch with a identifier (SHA) of 3b946c4
  • 1 file was changed (added), in this case it was .gitignore
  • The creation mode of the added file was 100644. The create mode is an octal representation of the file mode. (Click the down arrow to understand the various types of creation modes.)

Git Create Modes

Octal Value Mode
040000 Directory
100644 Regular, non-executable file
100664 Regular, non-executable, group-writeable file
100755 Regular executable file
120000 Symlink
160000 Gitlink

git log

Now that we made an initial commitment to the Git repository, we can view a log history of the repository.

This repository only has one commit so far, so we only see one. Above we can see the complete, 40-character SHA-1 checksum hash. You can see the author of the commit (assuming they have properly set up their .gitconfig file in their home directory), the datetime at which the commit was made, and the message assigned to the commit (if one exists).
Click the down arrow to view other strategies for viewing logs.

Formatting Git Logs

Git logs can be useful in determining the history of commits, including what files changed, who changed them, and why. A number of formatting flags can be used. The short table below shows some useful flags, but the entirety of the flags can be found using the link.

Flag Intent
--name-status Shows the names and statuses of files changed
--graph Draw a text-based graphical representation of the commit history
--abbrev-commit Shorten the commit's SHA
--pretty=oneline Condense the log message down to a single line

git status

Let’s create some new files and a directory. Here we create a logic file for the overall droid system. We also create a directory for communication systems of the droid, in which we will place a translator file to translate speech input.

Using git status, we can see what files Git see have changed or been added.

git status

Consider this segment from the git status message. It indicates that these files have not yet been staged.

nothing added to commit but untracked files present (use "git add" to track)

Git staging is the intermediary step between modifying files and committing them. The "stage" stores information about what will go into the next commit.

Let’s stage these files using git add

git add

To stage files, we use the git add command. In this instance, we added all files in and under the current directory using the command: git add .

After adding these files to the Git staging area, we can see with git status that these files are now considered “Changes to be committed” and are highlighted in green.

Select the down arrow for various formats for adding files using git add

Formats for Adding Files

The git add command can stage files and directories in a number of ways. The table below provides a few relevant examples.

git add . Stages files under the current directory
git add <file1> <file2> ... <fileN> Stage specific files
git add <dir1> <dir2> ... <dirN> Stage specific directories
git add <file1> <file1> <dir1> Stage both specific files and directories

git commit

We’re ready to commit these changes, so let’s do so. We use the git commit command to commit all the staged files. It is very good practice to use the –m flag with git commit to include a brief message detailing the purpose or summary of the commitment.

Checking the status of the repository again, we see that the tree is in a clean state.

git diff

Another important feature of Git is it’s ability to show the changes within files since the last commit. We have already seen the list of files added or modified using git status, but we can view the specific changes using the git diff command.

In this example, we have created a new, empty directory called power. We have also made a modification to the logic file. Running git status only shows the modifications to the logic file – why? Git only recognizes directories if they are NOT empty.

git diff

Here we have run git diff. Since we have only changed a single line of a single file, the result of this diff is rather simple, but it does a good job at illustrating the purpose of git diff.

The response contains details such as the file that was changed and the exact lines that were changed. As expected, we see the changed file is logic.py. The already-committed version of the file is donated by three dashes (---). The modified file is denoted by three plus signs (+++). The modified or added lines are shown in green, whereas the previous lines are shown in red.

git reset

Now let’s say we went ahead and added the logic file to the staging area of Git.

But wait! We realized before committing that we do NOT want to make these changes. We could continue to change the file without committing it, but for the sake of example, let’s unstage the file for safety.

git reset

We can unstage a file, directory or multiple of both using the command:
git reset HEAD <files or directories>
The term HEAD is a reference to the last commit of the currently checked-out branch.
Checking the repository’s status afterwards shows that the logic.py file is now unstaged.

git reset

WARNING

Now that I’ve shown you about git reset, you may happen to stumble upon the following command when searching the Internet.

git reset --hard HEAD

BE CAREFUL

This command will destroy all of the uncommitted changes you have made. There are some instances where it is useful or even necessary, but it should be used with caution and awareness.

git checkout -- <file>

A Reset Alternative

This command will erase any changes you have made to the specified file (or files, if a directory is specified) and checkout the most recent version of the file or files.

git reset updates the staging area, whereas
git checkout -- updates the working directory.

git checkout (without the ”--”) is also used to switch between branches.

git rm

It is rare that you will ever have to use this, but it’s worth a brief mention. git rm removes a file from both your system and the Git index of that repository.

In the example above, we remove the logic file using git rm. Notice that both Git and the computer system realize the file logic.py has been deleted. This deletion can be reverted by using git reset --hard HEAD, but of course this should be used with caution, as previously mentioned.

Note that this does not remove it from the history of the repository.

Summary of Git Commands for Local Use
git status git reset
git diff git commit
git add git rm
git log

We’ve covered a number of useful Git commands.

Other Useful Git Commands
git branch git fetch
git checkout git pull
git push git merge
git log

These Git commands above are relevant to projects that have network connectivity. While some of these, such as branch and checkout can be used without network connectivity, they’re most-applicable to projects that are hosted on a server.
